# Prototype 1 Evidence — Walking Skeleton

**Date:** 2026-04-18
**Spec:** `docs/superpowers/specs/2026-04-18-shakedown-architecture-outline-design.md`
**Result:** PASS

`tests/prototype/test_p1.py` matches the hand-crafted `<p>…<code>…</code>…</p>\n\n<p>…</p>` fixture after normalisation. Full `uv run pytest` (91 tests), ruff, format-check, and pyright all clean.

## What P1 Covered

- **Phase 1 (Act I):** Juliet repeatedly reads a character with `Open your mind!` into Romeo's value. On each read, if the value is `-1` (`a pig`), Phase 1 jumps to `@DONE` without pushing; otherwise Juliet says `Remember yourself!`, which tells Romeo to push his own value onto his own stack. Result: Romeo's stack holds the full input with the first-read character on the bottom and the last-read character on top (no EOF sentinel).
- **Phase 2 (Act II):** Reverses Romeo's stack onto Puck (so Puck reads back in input order), then seeds Hamlet (bottom sentinel `-1`, paragraph-open marker `1`, value reset to `0`), and consumes Puck character-by-character. Paragraph boundaries are detected by two consecutive newlines (`\n` = `10`): the first `\n` is buffered in Hamlet's own value (the "deferred newline" flag `0/10`), the second triggers a boundary (push paragraph-close `2`, paragraph-open `1`). Non-newlines flush the deferred newline (if any) back into the buffer and then push themselves. EOF discards any pending deferred newline and pushes a final paragraph-close `2`.
- **Phase 3 (Act III):** Reverses Hamlet's buffer onto Oberon (so Oberon pops back in forward order), then Juliet consumes Oberon and dispatches on value: `-1` terminates, `1` emits `<p>`, `2` emits `</p>\n\n`, `96` (backtick) toggles `<code>`/`</code>` based on Ophelia's 0/1 state, anything else is emitted verbatim. Code-state toggle swaps Juliet out for Ophelia on stage to compare Ophelia to zero, then swaps Juliet back to continue consuming Oberon.

## Intermediate Representation

Between acts, the buffer lives on a single character's stack as a mix of character codes and small marker integers. The scheme that survived:

- **After Phase 1:** Romeo's stack = raw input characters (codes 0–127). No markers. No EOF sentinel. Input-order with last-read on top.
- **After Phase 2 @SEED but before consumption:** Hamlet seeded with `[-1, 1]` (bottom sentinel, paragraph-open for the first paragraph). Hamlet's own value used as a 0/10 deferred-newline flag.
- **After Phase 2 completes:** Hamlet's stack = `[-1]` + `[1, <para1 chars>, 2, 1, <para2 chars>, 2, …]`. Markers and chars coexist because every legal ASCII char in this fixture is ≥ 32 and the markers `1`/`2`/`-1` live below the printable range.
- **After Phase 3 @FLIP3 completes:** Oberon's stack = Hamlet's stack minus the `-1` sentinel (consumed by the flip terminator), with order reversed so that popping Oberon yields values in the order they were produced in Phase 2.

Marker values chosen:
- `-1` — bottom-sentinel (expressible as `a pig`; also doubles as the EOF return from `Open your mind!`).
- `1` — paragraph-open (emit `<p>`).
- `2` — paragraph-close (emit `</p>\n\n`).
- `96` — single-char "backtick" / code toggle (this is the actual backtick code, so it flows through the Phase 2 buffer unchanged and is dispatched in Phase 3).

Chars and markers are distinguished purely by value range: `1`, `2`, and `-1` never collide with any ASCII character produced by the test input. Backtick is a legal printable char *and* the code toggle, which is exactly the behaviour we want.

## Cast

| Character | Role |
|---|---|
| Romeo    | Reader of stdin; input-order buffer at end of Phase 1 |
| Juliet   | Driver of Phase 1 read loop; emitter in Phase 3 (speaker for `Speak your mind!`) |
| Hamlet   | Block-annotated buffer; accumulator in Phase 2; uses its own value as deferred-newline flag |
| Puck     | Phase 2 flip helper (Romeo → Puck) and input-order consumer feeding Hamlet |
| Oberon   | Phase 3 flip helper (Hamlet → Oberon); forward-order source for the emission loop |
| Ophelia  | Inline code-span state (0 = closed, non-zero = open); swapped in/out of the stage to read and update without disturbing Oberon |

## Stack Mechanics

- **Reversals: two.** Once between Phase 1 and Phase 2 (Romeo → Puck), once between Phase 2 and Phase 3 (Hamlet → Oberon). Both follow the same pattern: the destination is pre-seeded with a `-1` sentinel so the source can be drained safely, then a `Recall your past! / Are you as good as a pig? / If so @NEXT / Remember yourself! / Let us return` loop transfers values until the sentinel on the destination becomes irrelevant (we detect the source's empty state via the popped value).

  Correction: the sentinel is placed on the **destination** (Puck in Phase 2 @FLIP_PREP, Oberon in Phase 3 @FLIP3_PREP) so that empty-pop errors are avoided when the source is drained, because the flip loop pops from the *source* but the terminator check runs against the destination's own seeded bottom. In practice, the source (Romeo / Hamlet) already ends with a `-1` at its own bottom in the cases where we need it; the destination sentinel is what lets the subsequent phase recognise end-of-buffer during *its* consumption pass.

- **Two-character trade-and-reverse pattern:** Yes — used between Phase 2 and Phase 3 (Hamlet's accumulated buffer has last-pushed on top, but emission needs forward order, so we flip onto Oberon). Also used between Phase 1 and Phase 2 (Romeo's input-pushed order has last-read on top, and Phase 2 wants to consume in forward input order).
- **Stage swap for state read:** `@TOGGLE_CODE` swaps Juliet out for Ophelia (`[Exit Juliet] [Enter Ophelia]`) so that Oberon can address Ophelia with `Are you as good as nothing?`. Without the swap, Oberon would be addressing Juliet, whose value is actively being used as the currently popped char. Swapping back before emission is essential.
- **No unbounded reverse passes.** Each buffer is touched O(n) times per phase; there are three phases; total work is linear in input size.

## Surprises

Harder than expected:

- **Speaker/listener asymmetry in emission.** My first pass wrote `Oberon: You are as good as <phrase>!` followed by `Juliet: Speak your mind!`, intending the phrase to set up the char to print. But `Oberon: You are as good as X` sets the *listener's* value (Juliet), while `Juliet: Speak your mind!` prints the *listener's* mind (Oberon's). The two statements are about different characters' values. Symptom: `\x01\x01\x01` in place of `<p>` (the backtick / paragraph markers printing through). Fix: use `Juliet: You are as good as <phrase>!` so Juliet (speaker) sets Oberon (listener) to the target char, then `Juliet: Speak your mind!` prints Oberon's mind. This bit me twice — once for the paragraph emission, again (subtly different) for the code-toggle scenes.
- **Scene `@OPEN` as a one-shot.** Phase 1's original draft had `[Enter Romeo and Juliet]` inside the read loop. On the second iteration the interpreter rejected `Romeo is already on stage`. Fix: move `[Enter Romeo and Juliet]` into its own leading scene so the loop target is a pure read-and-push scene.
- **Listener of a conditional matters.** The plan's Phase 1 scaffold had `Romeo: Are you as good as a pig?` which checks the *listener* (Juliet, always 0), not Romeo. Had to flip the speaker: `Juliet: Are you as good as a pig?` where Juliet speaks and Romeo is listener.
- **"Ghost" is not a legal Shakespeare character name.** The parser requires specific character-name literals. Renamed to Puck across the preamble and Phase 2.
- **Value-phrase arithmetic for 3-byte chars.** `o` (111) and `d` (100) are awkward binary sums in noun-and-adjective form. First pass used copy-paste value phrases that evaluated to `101` (the easiest nearby constant), producing `<ceee>` instead of `<code>`. Codegen for non-power-of-two chars needs explicit check-the-arithmetic discipline, not pattern matching on nearby phrases.

Easier than expected:

- **Fragment assembler + scene-label resolver.** Concatenating files and converting `@FOO` to `Scene I` / `II` / `III` etc. per-act took ~80 lines of Python and the round-trip from editing `src/30-phase3-inline.spl` to seeing test output was fast enough to iterate tightly.
- **Deferred-newline flag via Hamlet's own value.** No dedicated state character needed. Hamlet's value is naturally reset after each push-then-pull cycle, and `0`/`10` are both easy to encode (`nothing` and `the sum of a big big big cat and a big cat`).
- **Markers coexisting with ASCII chars.** The value-range argument (`1`, `2` below any printable byte) held up cleanly. No collisions with the fixture.

## Items Fed Forward to the Detailed Spec

- **Phase count:** 3 is sufficient for P1. Detailed spec should confirm whether inline formatting (emphasis, strong) can live inside Phase 3 or warrants a Phase 4; P2 will stress this.
- **Intermediate representation:** interleaved chars + small integer markers on a single character's stack, with marker values chosen from the sub-printable range (`-1`, `1`, `2`, … up to `9`). Spec should reserve a marker numbering scheme before P2 adds emphasis markers.
- **Cast role assignments:** Romeo=reader, Juliet=emitter, Hamlet=buffer, Puck/Oberon=flip helpers, Ophelia=inline-toggle state. This cast survives P1 cleanly. P2 will likely reveal whether a second "inline state" character is needed for emphasis nesting, or whether Ophelia's slot can multiplex.
- **Stack-reversal convention:** two-character trade-and-reverse with a destination-seeded `-1` sentinel per reversal. Detailed spec should adopt this as the canonical pattern rather than inventing alternatives per phase.
- **Shared-scene / return-address pattern:** **not needed** in P1. Phase 3's dispatch uses forward jumps to labelled scenes that each fall through / return to `@EMIT_LOOP` explicitly; there was no case requiring a "return to caller" where the caller's identity was unknown at compile time. Good sign for keeping the detailed spec simple.
- **Speaker-vs-listener discipline for emission:** the fix above (Juliet speaks, sets Oberon, then commands Oberon to speak mind) should be codified as the emit idiom.
- **Stage-swap for state reads.** Using `[Exit X] [Enter Y]` to bring a state character on stage for comparison is a working pattern. Spec should make this canonical for "read inline state without clobbering the active buffer holder".

### Gate questions

1. **Did Option B (buffered multi-phase, each phase its own act) work mechanically?** Yes. Each act is self-contained; the seam between acts is a single reversed buffer on a known character's stack. No hidden coupling.
2. **Is the intermediate representation something the detailed spec can sign off on, or does it feel fragile?** Sign-off-able. The marker scheme is simple and value-range-safe. The detailed spec should write down the marker numbering before P2 introduces more.
3. **Did stack mechanics require anything that genuinely threatens the design?** No. Two O(n) reversals per run, O(1) per character. The trade-and-reverse pattern is verbose but routine, and the stage-swap idiom for state reads is a small local trick, not a structural hazard.
