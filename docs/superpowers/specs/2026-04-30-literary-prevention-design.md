# Literary Prevention Design

## Goal

Prevent future SPL and codegen work from reintroducing invented inline literary
surfaces, monotonous numeric prose, or run-loop prompts that omit the literary
rules.

The current cleanup makes existing production SPL conform. This design makes the
next agents' path harder to get wrong by moving controlled prose into
`src/literary.toml`, teaching assembler/codegen to consume that data, and adding
prompt-level gates for future run-loop work.

## Scope

In scope:

- `scripts/assemble.py`
- `scripts/codegen_html.py`
- shared literary-surface loading code under `scripts/`
- `src/literary.toml`
- active run-loop prompts such as `docs/prompt-shakedown.md`
- future implementation plans under `docs/superpowers/plans/`
- repo guidance in `CLAUDE.md` / `AGENTS.md`
- tests that enforce the contract above

Out of scope:

- changing Markdown behavior
- changing the selected runtime architecture
- generating the whole SPL program from a semantic DSL
- rewriting historical probe programs in `docs/spl/probes/**`
- replacing human review of taste with a fully subjective style checker

## Chosen Approach

Use a narrow TOML-backed placeholder layer in otherwise readable SPL fragments,
plus prompt-author enforcement.

Production fragments remain recognizably hand-written SPL. The assembler does
not become a broad DSL renderer. Instead, fragments may contain explicit
placeholders for controlled literary surfaces:

- play title
- act titles
- scene titles
- Recall text
- recurring value phrases
- other named decorative or stable utility surfaces only when the spec declares
  them controlled

Codegen uses the same literary data for generated byte/value phrases. Prompt
authors and run-loop agents are separately required to include the literary
protocol whenever a plan or prompt asks for SPL, assembler, or codegen changes.

This keeps Markdown behavior in SPL and keeps the source readable, while making
the reusable literary surfaces data-driven instead of memory-driven.

## Data Model

`src/literary.toml` is the source of truth for controlled surfaces.

Required sections:

- `[play]` with the locked play title.
- `[acts.<act_key>]` with locked act titles.
- `[scenes.<SCENE_LABEL>]` with title, act, pattern, and optional iconic or
  dramatic tags.
- `[characters.<character>.stable_utility]` for recurring signed values.
- `[characters.<character>.soft_variation]` for character-specific legal
  variants.
- `[characters.<character>.recall_pool]` for Recall descriptions available to
  that speaker.
- `[value_atoms.<family>]` for numeric atom families used by codegen and
  production placeholders.
- `docs/superpowers/notes/spl-literary-protocol.md` as the reusable prompt
  block listing required documents and verification commands for SPL-changing
  prompts.

The implementation may add fields while writing the plan, but it must not rename
or remove the required sections above without a follow-up design amendment. It
must preserve these ownership rules:

- human-authored prose lives in TOML
- assembler/codegen select or substitute prose; they do not invent it
- symbolic keys, not prose strings, appear in source wherever a controlled
  surface can be represented without harming readability

## Placeholder Surface

The assembler supports a small, explicit placeholder syntax. Placeholders use
the `@LIT.<section>.<key>[.<field>]` form so they are grep-friendly and
unambiguous:

- `@LIT.play.title`
- `@LIT.acts.act1.title`
- `@LIT.scenes.DETAB_RAW.title`
- `@LIT.recall.hecate.detab_cauldron`
- `@LIT.value.prospero.byte_amp`

Assembler behavior:

- resolve all placeholders before writing `shakedown.spl`
- fail with a clear error on unknown keys
- fail if any `@LIT.` placeholder remains unresolved
- preserve the existing scene-label resolution behavior
- keep generated output deterministic

The placeholder layer is intentionally not a control-flow language. It may
substitute text, but it does not decide which scene to enter, which stack action
to perform, or which Markdown branch to take.

## Codegen Behavior

`scripts/codegen_html.py` stops hardcoding numeric prose atoms like
`a big big big big cat`.

Instead, it loads a validated atom family from `src/literary.toml` and owns the
recipe selection for larger values:

- atom values must preserve SPL numeric semantics
- atom text must pass the atom-length cap from the literary spec
- generated phrases must remain parser-accepted
- decomposition is deterministic
- repeated same-atom sum chains are forbidden for large values
- compact verified arithmetic forms such as `the square of A` and
  `the product of A and B` are preferred over repeated addition when they reduce
  repetition without changing value
- `parse_value_phrase` understands the configured atom phrases so tests can
  still round-trip generated values and compact arithmetic recipes

For generated byte literals, codegen may use a shared default family or a
speaker-specific family when TOML provides one. Critical values may remain
canonical where recognizability matters, but the decision is represented in
TOML rather than buried in Python constants.

Atom families are data; recipe selection is codegen policy. `src/literary.toml`
names legal value surfaces, while `scripts/codegen_html.py` decides whether a
value is best expressed as one atom, a bounded sum, a product, or a square. The
policy must be covered by tests for representative large values, not only by
round-tripping small atoms.

## Prompt And Agent Protocol

Future prompt authors must not rely on memory of the literary rules.

Add a concise SPL literary protocol to `CLAUDE.md` / `AGENTS.md`:

- prompts or plans that edit `src/*.spl`, `scripts/assemble.py`, or codegen
  must require reading `docs/spl/literary-spec.md`,
  `docs/spl/style-lexicon.md`, `docs/spl/codegen-style-guide.md`, and
  `src/literary.toml`
- prompt authors must use the reusable protocol block for SPL-changing
  run-loop prompts
- controlled surfaces must route through TOML-backed placeholders or codegen
  loaders once those mechanisms exist
- new recurring prose belongs in TOML first, then source references it by key
- implementation agents must run the literary compliance tests named by the
  active plan

Add a reusable prompt block under `docs/superpowers/notes/` or
`docs/superpowers/templates/`. Active run-loop prompts reference that block or
include its required text verbatim.

The protocol should tell implementing agents to classify new prose before
writing it:

- Critical
- Stable Utility
- Incidental
- Recall
- title or scene title
- dramatic tag

The classification decides whether TOML, codegen, or hand-authored SPL owns the
surface.

## Automated Gates

Add tests that make the protocol enforceable.

Assembler/codegen gates:

- `assemble.py` resolves known literary placeholders.
- unresolved or unknown placeholders fail.
- generated `shakedown.spl` contains no `@LIT.` placeholders.
- codegen loads configured value atoms from TOML.
- configured atom families round-trip through `emit_byte` and
  `parse_value_phrase`.
- production source has no oversized `big ... cat` atom.
- generated and production value recipes do not repeat the same high-value atom
  more than a small fixed limit in a single statement.
- generated large values use compact verified recipes for representative values
  such as 38, 65, 97, 256, and the current Slice 1 output length.

Prompt gates:

- active run-loop prompts that mention SPL/codegen implementation must include
  or reference the reusable SPL literary protocol
- in-flight implementation plans that edit `src/*.spl`, `scripts/assemble.py`,
  or codegen must mention the required literary docs and compliance tests
- `CLAUDE.md` / `AGENTS.md` must point prompt authors at the protocol

These tests are intentionally structural. They cannot judge beauty, but they can
prove future agents were given the right inputs and that controlled surfaces are
not bypassing the data path.

## Error Handling

Failures should be early and specific:

- missing TOML section: name the section
- unknown placeholder: name the placeholder and source file
- invalid atom value: name the family and value
- atom phrase over cap: name the phrase and TOML key
- unparseable generated recipe: name the generated phrase and source value
- prompt missing protocol: name the prompt or plan and the missing requirement

The assembler and codegen should raise exceptions rather than silently falling
back to hardcoded prose. Silent fallback would recreate the original failure
mode.

## Rollout

Implement in small, behavior-preserving steps:

1. Add the prompt protocol text and prompt/plan compliance tests.
2. Add a shared TOML loader for literary surfaces.
3. Teach codegen to load value atom families from TOML.
4. Add assembler placeholder resolution for titles and scene titles.
5. Extend placeholders to Recall and recurring value phrases.
6. Convert existing production fragments to placeholders where readability is
   preserved.
7. Keep Markdown parity and current fixture behavior passing throughout.

The cleanup branch may land before this prevention work begins. Prevention
should start from the cleaned production SPL so tests lock the desired final
surface rather than the currently non-compliant source.

## Validation

Required verification for the implementation plan:

- `env UV_CACHE_DIR=/tmp/uv-cache uv run pytest`
- focused assembler tests for placeholder success and failure
- focused codegen tests for TOML-loaded atoms and round-trips
- prompt compliance tests for active run-loop prompts and in-flight plans
- `env UV_CACHE_DIR=/tmp/uv-cache uv run pytest tests/test_mdtest.py -k 'Amps and angle' -q`
- `bash -n shakedown`
- `env UV_CACHE_DIR=/tmp/uv-cache uv run pyright`
- relevant `ruff` checks for touched Python files

## Non-Goals

This design does not make every line of SPL generated. It does not try to
automate subjective taste. It does not remove the need for human review of
literary choices.

It does make the expected path explicit, data-backed, and testable so future
agents have to deliberately bypass the system to reintroduce the same class of
mistake.
